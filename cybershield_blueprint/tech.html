05.1

SOC Design Philosophy: Automation-First

Core Architectural Insight

Run the SOC automation-first. Playbooks close repeatable alerts. Analysts focus on high-impact exceptions. This model keeps quality high at SMB pricing.

Traditional MSSP Model

Analyst-Heavy

  • 1 analyst per 8–12 clients
  • Manual alert triage: 60–70% of time
  • High salary overhead
  • Limited scalability
  • SIEM licensing: $50K+/year (Splunk/QRadar)
  • Break-even: 30–40 clients minimum

Our Model

Automation-First
  • 1 analyst per 30–35 clients
  • 85% alerts auto-resolved/enriched
  • Low salary overhead (Jordan-based)
  • Sub-linear cost scaling
  • SIEM: Wazuh + Elastic (infra cost only)
  • Break-even: 20–22 clients
Result

Structural Advantage

  • 73–79% gross margin per client
  • vs. industry average 40–55%
  • Analyst time: 2–5h/client/month
  • vs. industry 15–25h/client/month
  • Scales to 80+ clients with 4–5 staff
  • Traditional MSSP needs 10+ analysts at 80 clients

The Three Engine Layers

Layer 1 — Detection (Wazuh + Elastic): Collect, normalize, and correlate logs. Produce scored alerts with tenant context.

Layer 2 — Automation (Shuffle SOAR): Enrich, classify, and execute standard response steps. Escalate only high-severity or ambiguous events.

Layer 3 — Advisory (Analyst + vCISO): Make incident decisions, interpret compliance impact, and communicate to clients.

Synapse Core — Internal Platform Identity

Synapse Core is one operating platform, not a loose toolset. Standardization keeps onboarding fast and delivery consistent.

Core stack: Wazuh + Shuffle + Elastic + TheHive + OpenCTI + internal rule/playbook libraries. This stack drives the 85% automation target and protects gross margin.

Long-term optionality: white-label Synapse Core to other Jordan IT firms as a platform-as-a-service revenue stream (JOD 500–800/month per partner).

Business effect

  • Cost impact: Open-source-first architecture keeps fixed platform costs low while clients scale.
  • Analyst time impact: Automation-first operations move analysts from repetitive triage to exception handling.
  • Margin impact: Lower delivery cost per client supports 73–79% gross margin at target scale.

05.3

Multi-Tenant Architecture — Client Isolation

Critical Security Requirement — Never Shared Indexes

In a multi-client MSSP, your platform is a prime target. Any cross-tenant data exposure is a catastrophic breach. Enforce strict tenant isolation from Day 1.

Elasticsearch Multi-Tenant Design

  • Index pattern per client: wazuh-alerts-client-acme-* — never shared indexes
  • Index lifecycle policy (ILM): Hot → Warm → Cold → Delete based on retention tier
  • Role-based access (RBAC): Each client namespace has its own Elastic role — analysts cannot query other namespaces without explicit grant
  • Kibana Spaces: Separate dashboard space per client with client-specific visualizations
  • Data retention by tier: SOC Lite: 30-day hot + 60-day cold. SOC Standard: 60-day hot + 6-month cold. SOC Pro: 90-day hot + 12-month cold

Wazuh + TheHive Isolation

  • Wazuh manager groups: Each client is a named group in Wazuh manager. Agents tagged with group ID. Rules and decoders can be group-specific.
  • TheHive organizations: One org per client. Analysts assigned to specific orgs only. Cases, observables, and timeline never cross orgs.
  • Shuffle workflows: Named workspace per client. Client-specific API keys for integrations (firewall, AD, email). No shared workflow handles multiple clients.
  • Network segmentation: Client agent traffic flows via encrypted channel (Wazuh protocol over port 1514/1515). No client can reach another client's agent.

Admin Security Model (Protecting Your Own Infrastructure)

Access Control:

  • All admin access via YubiKey hardware MFA (mandatory)
  • No SSH password auth — key-based only
  • Admin VPN required for all infra access
  • Separate admin account per analyst — no shared credentials
  • Privileged Access Workstation (PAW) policy for admin tasks

Monitoring Your Own Infra:

  • Wazuh agent deployed on all Hetzner servers (monitor yourself)
  • Fail2ban on all public-facing SSH
  • Cloudflare Zero Trust for admin panel access
  • Weekly automated vulnerability scan of own infra
  • Annual external pentest of own infrastructure (mandatory)

Incident Response for Own Infra:

  • Dedicated runbook for "MSSP own infra breach" scenario
  • Client notification procedure: 4h SLA for breach disclosure
  • Cyber liability insurance: JOD 600–1,000/year (mandatory Day 1)
  • Backups tested monthly — restore drill quarterly
  • SOC 2 Type II target for Year 3 (builds enterprise trust)

Business effect

  • Cost impact: Isolation-by-design reduces breach blast radius and lowers potential legal/remediation costs.
  • Analyst time impact: Tenant-scoped roles and workspaces cut investigation friction and reduce context-switching.
  • Margin impact: Strong tenant isolation increases contract trust and compliance confidence, improving retention and enterprise win rates.

Automation Rate Achievement: How to Reach 85%

In a typical 100-alert week: ~50 auth-noise alerts map to PB-002/PB-009, ~20 phishing alerts map to PB-001, ~10 vuln alerts map to PB-005, and ~5 asset alerts map to PB-011. Result: 85 alerts auto-resolved, 15 escalated. As tuning improves in months 1–3, false positives drop and automation stays stable.

Business effect

  • Cost impact: Higher automation lowers cost per alert handled and limits headcount growth.
  • Analyst time impact: Higher automation rate directly increases analyst capacity (target 30–35 clients per analyst).
  • Margin impact: Stable auto-resolution rates preserve service quality while expanding gross margin at scale.

Log Economics — The Critical MSSP Margin Driver

Why Unlimited Logging Will Destroy Your Margins

A verbose IIS/Apache server can generate 50–200 GB/day. At Sentinel pricing ($2.46/GB), that is $3,700–$14,760/month for one client. Even on Hetzner, uncontrolled logs make compute and storage costs volatile. Contract log caps are mandatory from Day 1.

Pre-Filtering Pipeline (Reduces Effective Volume by 40–60%)

Stage 1: Filebeat Pre-Filter

At the client endpoint/server, Filebeat uses drop_fields and drop_event. Forward only security-relevant events to SIEM. Keep raw logs on-client per contract retention. Typical result: 50–70% less transfer and SIEM storage.

Stage 2: Wazuh Decoder + Rule Filter

Wazuh applies decoders and rules before alert generation. Only events with severity ≥ 3 create alerts. Deduplication counts identical events once per 60 seconds. This removes repetitive noise before storage.

Stage 3: Elasticsearch ILM

Elasticsearch ILM enforces tiered retention: Hot 30–90 days, Warm 90–180 days, Cold 180–365 days, then Delete by SLA. This keeps active compute low and maintains compliance retention.
